# Chapter 05 (교재)

## Item 43 any는 가능한 좁은 범위에서만 사용하기

- 함수 전체나 변수 선언 시 any를 쓰지 말고, 구체적인 표현식에만 단언문(as any)를 사용

## Item 44 any를 구체적으로 변형해서 사용하기

- 그냥 any라고만 쓰지 말고 최소한의 정보라도 제공하기
- any 보다는 any[], {[key: string]: any}, (...args: any[] => any)

## Item 45 함수 안으로 타입 단언문 감추기

- 내부 로직이 복잡해서 타입을 맞추기 어려울 때는 함수 내부에서 any나 단언문을 사용하고, 외부로 노출되는 인터페이스(반환 타입)은 정확하게 정의하기

## Item 46 타입을 모르는 경우 any 대신 unknown 사용하기

- any는 모든 타입의 하위 타입이자 상위 타입이지만, unknown은 오직 상위 타입이기만 함
- 사용자가 타입을 좁히도록 강제하기 때문에 훨씬 안전함

# Chapter 06 (교재)

## Item 50 제네릭이 타입 간 함수라고 생각하기

- 제네릭은 타입을 입력받아 새로운 타입을 출력하는 함수와 같다.
- Pick<T, K>는 T와 K라는 타입을 인자로 받아 새로운 객체 타입을 반환하는 "타입 함수"

## Item 51 불필요한 타입 매개변수 지양하기

- 타입 매개변수를 쓰지 않고도 타입을 정의할 수 있다면 쓰지 않는것이 좋음

## Item 52 오버로딩 타입보다는 조건부 타입 사용하기

- 동일한 함수가 여러 타입을 처리해야 할 때, 함수 오버로딩을 여러 줄 쓰는 것 보다 조건부 타입 하나를 쓰는 것이 더 정교함
- 오버로딩은 각 선언을 독립적으로 처리하므로, 조건이 복잡해지면 의도치 않은 타입으로 추론될 때가 있음

# 가장 인상 깊었거나 이해가 안 갔던 내용

## 인상 깊었던 내용

- **Item 46**
- 많은 개발자들이 귀찮다는 이유로, 혹은 당장 에러를 끄기 위해 any를 남발하지만 unknown은 겸손한 any임
- any는 타입 시스템을 파괴하지만, unknown은 타입 시스템을 유지함
- unknown은 "데이터가 들어올 수 있지만, 확실히 검사하기 전까지는 사용할 수 없어"와 같음
- API 응답값 처리나 외부 라이브러리 연동 시 unknown을 기본으로 사용하는 습관을 들이면, 런타임 에러의 90% 이상을 컴파일 타임에 잡아낼 수 있음

## 더 공부해야할 내용

- 타입스크립트 유틸리티 타입

1. 객체의 가시성 및 변형

- Partial<T> = 모든 속성을 optional로 변경
- Required<T> = 모든 optional을 필수로 변경
- Readonly<T> = 모든 속성을 읽기 전용으로 변경

2. 특정 속성만 골라내기

- Pick<T, K> = T에서 K속성만 골라냄
- Omit<T, K> = T에서 K속성만 제외함

3. 집합과 매핑

- Record<K, T> = 키가 K이고 값이 T인 객체를 만듬
- Exclude<T, U> = T 유니온 타입에서 U와 겹치는 타입을 제외함
- Extract<T, U> = T 유니온 타입에서 U와 겹치는 타입만 추출함

4. 함수 관련

- ReturnType<T> = 함수의 반환 타입을 가져옴
- Parameters<T> = 함수의 매개변수 타입들을 튜플로 가져옴
